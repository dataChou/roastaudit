### 失败原因分析

Step 2/3/4 全部返回 404，根因：**本地环境无 Upstash 配置，demo 模式的 `storeReport()` best-effort 存储静默失败，后续 checkout 和 report-pdf 从 Upstash 读取不到报告**。

具体链路：
1. `audit.js:67` — `storeReport()` 调用 `upstash()` → `upstash()` 检测到 `!url || !token` → 返回 `null` → 存储失败但被 `try/catch` 吞掉
2. `checkout.js:35-37` — `upstash('GET', ...)` → 返回 `null` → 判定 `!result` → 返回 404 "Report not found"
3. `report-pdf.js:34-37` — 同理 → 返回 404 "Report not found or expired"

**这是 demo 模式设计的关键缺陷：spec 规定了 store 是 best-effort，但没有为 checkout/report-pdf 提供 demo 模式下的 fallback 路径。当 Upstash 不可用时，完整的 demo 流程（audit → checkout → pdf）无法走通。**

## 发现的问题

### BUG-1: L3 E2E Step 2-4 在无 Upstash 时全部失败（严重）

- **影响**: 本地开发者没有 Upstash 配置时，demo 模式的完整流程无法走通
- **根因**: `audit.js` 的 demo 分支使用 best-effort Upstash 存储，但 `checkout.js` 和 `report-pdf.js` 没有对应的 demo fallback
- **修复建议**:
  - 方案 A（推荐）: checkout.js 和 report-pdf.js 在 demo 模式下（`reportId` 以 `demo-` 开头）使用内存 Map 作为 fallback 存储
  - 方案 B: audit.js 的 demo 分支改用内存 Map 存储（而非 best-effort Upstash），checkout/report-pdf 在 demo 模式下也查内存 Map
  - 方案 C: test-server.js 内置一个内存 Upstash mock（拦截 upstash 函数调用）

### BUG-2: report-pdf.js 对 demo 报告的 fullReport 字段缺失

- **影响**: 即使 Step 2-3 修复后，Step 4 的 PDF 生成会 crash（`parsed.fullReport` 为 undefined）
- **根因**: `audit.js:69-74` 返回给客户端的字段没有 `fullReport`，但存储到 Upstash 的数据中也没有 `fullReport`
- **修复建议**: demo 模式存储时加入 `fullReport: data.summary`（用 summary 充当 fullReport）

### NOTE-1: demo-flow.test.js 中 Gate 3.5 测试实际是空跑

- `report.js` 的 GET 端点返回了 200 但 body 中没有 lighthouse 数据（因为存储时就缺失），测试内的 `return` 跳过了断言
- 当前测试逻辑：`if (!res.ok) { console.log('⚠ ...'); return; }` → 实际上 res.ok 为 true 但数据可能不完整

### NOTE-2: `node --test tests/unit/` 目录参数报错

- 直接传目录会报 `Cannot find module`，需要用 `node --test tests/unit/*.test.js`
- 不影响测试结果，但与 process.md 中的命令不一致

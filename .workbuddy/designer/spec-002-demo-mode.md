# SPEC-002: Demo 模式设计

> 版本: v1.0 | 日期: 2026-06-14 | 作者: designer | 改动级别: L3 大改
>
> 背景: 沙盒网络隔离 + DEEPSEEK_API_KEY 可能无效，导致 L3 E2E 测试跑不起来。
> 目标: 在 `DEEPSEEK_API_KEY` 缺失时返回 mock 数据，L3 测试零外部依赖。

---

## 1. 触发条件

### 1.1 入口函数

新建文件 `api/_lib/demo-mode.js`，导出单一判断函数：

```javascript
// api/_lib/demo-mode.js
export function isDemoMode() {
  return !process.env.DEEPSEEK_API_KEY || process.env.DEEPSEEK_API_KEY === 'dummy';
}
```

### 1.2 覆盖的全部场景

| # | 条件 | 结果 |
|---|------|------|
| 1 | `DEEPSEEK_API_KEY` 未设置 | → 进入 demo 模式 |
| 2 | `DEEPSEEK_API_KEY` 为空字符串 `""` | → 进入 demo 模式 |
| 3 | `DEEPSEEK_API_KEY` 为 `"dummy"` | → 进入 demo 模式 |
| 4 | `DEEPSEEK_API_KEY` 为有效值（非空非 dummy） | → 正常模式 |
| 5 | `DEEPSEEK_API_KEY` 为有效值但 API 调用 401 | → **不降级**（正常报错，不静默切换） |

### 1.3 为什么是这三个条件

- **未设置** → `process.env.DEEPSEEK_API_KEY === undefined`，`!undefined === true`
- **空字符串** → `.env` 里写了 `DEEPSEEK_API_KEY=` 但没有值，`!'' === true`
- **`"dummy"`** → 明确标记值，开发者主动表达"我要跑 demo 模式"

三个条件覆盖了**无意识缺失**和**有意识选择**两种场景。

### 1.4 为什么不支持"API 调用失败后自动降级"

这是故意的设计决策：

- 自动降级会让生产环境静默返回假数据，用户花 $0.99/$1.99 拿到假报告 → 重大信任危机
- API 调用失败应该暴露（报错），而不是掩盖（demo）
- 开发者可以通过 `DEEPSEEK_API_KEY=dummy` 主动选择 demo 模式

## 5. 真实用户的影响（安全检查）

### 5.1 核心风险

如果生产环境（Vercel 部署后的 `NODE_ENV === 'production'`）`DEEPSEEK_API_KEY` 缺失：

- **没有 demo 模式检查**：产品静默返回 mock 数据，用户花 $0.99/$1.99 买到假报告 → 信任崩塌
- **有 demo 模式检查**：返回 500 报错，用户看到错误页面，但不会拿到假报告

### 5.2 安全策略（硬编码在 handler 里）

```javascript
if (isDemoMode()) {
  // 生产环境：拒绝 demo 模式，返回 500
  if (process.env.NODE_ENV === 'production') {
    console.error('FATAL: DEEPSEEK_API_KEY missing in production');
    return res.status(500).json({
      error: 'Service configuration error. Please contact support.',
    });
  }
  // 非生产环境：正常返回 mock 数据
  // ...
}
```

### 5.3 Vercel 部署侧

**建议**（不是强制，可选辅助手段）：
- 在 Vercel Dashboard → Environment Variables 中设置 `DEEPSEEK_API_KEY` 为必填
- 如果 API key 无效，Vercel 部署本身不会报错，但 `/api/audit` 会返回 500

### 5.4 安全矩阵

| 环境 | `DEEPSEEK_API_KEY` | `NODE_ENV` | 行为 |
|------|---------------------|------------|------|
| 本地开发 | 未设置 | `development` | → demo 模式（加 `[DEMO MODE]` 标记） |
| 本地开发 | `dummy` | `development` | → demo 模式 |
| 本地开发 | `sk-xxx`（无效） | `development` | → 正常模式，API 调用 401，报错 |
| Vercel Preview | 未设置 | `production` | → **500 Server Error**（禁止 demo 模式） |
| Vercel Preview | `dummy` | `production` | → **500 Server Error**（demo 标记值在生产也被拒绝） |
| Vercel Production | 未设置 | `production` | → **500 Server Error**（禁止 demo 模式） |
| Vercel Production | `dummy` | `production` | → **500 Server Error**（禁止 demo 模式） |
| Vercel Production | 有效值 | `production` | → 正常模式 |

### 5.5 为什么 Vercel Preview 也禁止 demo 模式

Vercel Preview Deployments 的 `NODE_ENV` 默认也是 `production`。如果有人在 Preview 部署中没设 `DEEPSEEK_API_KEY`，本应用会拒绝 demo 模式，返回 500——而非静默返回假数据。

## 6. 风险评估

### 风险 1：用户误以为假报告是真报告

- **概率**：中（如果用户看到了 `[DEMO MODE]` 标记但没注意到）
- **影响**：高（信任问题）
- **缓解**：
  - Mock summary 开头加 `[DEMO MODE]` 标记（已做）
  - Mock reportId 以 `demo-` 开头，方便后端日志区分
  - 前端可选：检测 reportId 以 `demo-` 开头时，顶部显示黄色 banner "This is a simulated report. No API calls were made."

### 风险 2：demo 模式被滥用（故意删 API key 省成本）

- **概率**：低（需要代码权限 + Vercel 环境变量修改权限）
- **影响**：高（用户拿到假报告）
- **缓解**：
  - 生产环境 `NODE_ENV=production` 时硬拒绝 demo 模式（已做）
  - Vercel Dashboard 强制要求 `DEEPSEEK_API_KEY` 环境变量（建议）
  - 审计日志（Vercel Logs）可追踪 API key 变更

### 风险 3：mock 数据和真实数据形状不一致

- **概率**：低（spec 明确定义了完整类型表）
- **影响**：中（前端渲染崩）
- **缓解**：
  - L1 单测 `demo-mode.test.js` 对每个字段做 `typeof` 断言（已做）
  - 前端对可选字段（如 `screenshotUrl`）应有 `?.` 安全访问
  - 曾有过真实 API 返回的所有字段都已映射到 mock

### 风险 4：NODE_ENV 在生产环境不是 "production"

- **概率**：极低（Vercel 标准行为，Node.js `process.env.NODE_ENV` 在 Vercel 部署中总是 `"production"`）
- **影响**：高（生产环境可能误入 demo 模式）
- **缓解**：
  - 在 `isDemoMode()` 里不仅判断 `NODE_ENV`，还判断 `VERCEL_ENV === 'production'`（Vercel 独有）
  - 或双重守卫：`if (process.env.NODE_ENV === 'production' || process.env.VERCEL_ENV === 'production')`

---

## 7. 推荐方案对比

### 方案 A：自动检测（推荐）

**触发**：`!process.env.DEEPSEEK_API_KEY || process.env.DEEPSEEK_API_KEY === 'dummy'`

| 维度 | 评价 |
|------|------|
| 使用体验 | 最佳——开发者本地 clone 后直接 `npm run dev` 就能跑，零配置 |
| 安全性 | 高——生产环境硬拒绝 + `[DEMO MODE]` 标记 |
| 维护成本 | 低——一个函数判断 + 一个 mock 生成函数 |
| 入侵性 | 低——只在 handler 开头加一个 `if` 分支，不动现有逻辑 |

**推荐理由**：默认场景（无 API key）自动启用，符合"零配置开发体验"原则。生产环境有硬守卫保护。

### 方案 B：手动开关 `DEMO_MODE=true`

**触发**：`process.env.DEMO_MODE === 'true'`

| 维度 | 评价 |
|------|------|
| 使用体验 | 差——需要额外设置环境变量 |
| 安全性 | 高（显式开关，不易误用） |
| 维护成本 | 低——与方案 A 类似 |
| 入侵性 | 低 |

**评价**：虽然安全，但开发者每次 clone 都要多一步 `export DEMO_MODE=true`，体验差。不是推荐方案。

### 方案 C：fetch 失败后自动降级

**触发**：DeepSeek/Upstash/Lighthouse 任意外部 API 调用失败后返回 mock 数据

| 维度 | 评价 |
|------|------|
| 使用体验 | 最差——用户看到假报告而不自知 |
| 安全性 | **极低**——生产环境静默返回假数据 |
| 维护成本 | 高——需要包裹每个 API 调用 |
| 入侵性 | 高——侵入多个模块 |

**评价**：风险最高，**不推荐**。

### 最终选择

**方案 A**（自动检测）为推荐方案。

备选：如果后续需要支持"用户在本地也有真实 API key 但要跑 demo 测试"，可以在方案 A 的基础上叠加 `DEMO_MODE=true`（方案 B 作为 override，优先级高于方案 A）。但当前 spec 不做这个——YAGNI。

## 一、改动分类（3 维度判定法）

任何改动必须先按以下 3 个维度分类，决定走哪条流程。

| 维度 | 小改 (L1) | 中改 (L2) | 大改 (L3) |
|------|------------|------------|------------|
| **用户感知** | 看不到 / 无关紧要 | 看到但不影响决策 | 新功能 / 定价 / 核心价值变化 |
| **数据流** | 不改 API / 不改 schema | 加字段 / 改接口 | 新数据源 / 新第三方 |
| **业务模型** | 不改商业模式 | 改文案 >3 处 | 改定价 / 改付费流程 |

**判定规则**：
- 3 维度任意 1 个 = L2，任意 2 个 = L3
- 判定不了 = 按上一级
- 文字 typo / 单字符 = L0（工匠自查，不需守门人）

## 三、Gate 3.5 轻量验证（L1/L2 改动专用）

L3 大改走完整 Gate 3（设计师出 PRD + 175 行验收报告）。

L1/L2 改动走 **Gate 3.5**（15-30 分钟）：

### Gate 3.5 必做 3 项

1. **工匠自测 demo 模式**（5 分钟）
   - `npm run demo-test` 全过
   - 或手动点一遍：输入 URL → 出报告 → 点 Unlock → 看全文 → 下载 PDF

2. **守门人跑 L1 单测**（5 分钟）
   - `node --test tests/unit/` 全过
   - 新加的代码必须有对应单测

3. **守门人跑 L3 E2E**（10 分钟）
   - `npm run demo-test` 全过
   - 若沙盒网络不通，交由用户本机跑并截图

### Gate 3.5 通过标准

- L1 单测 100% 过
- L3 E2E 全过（0 fail, 0 skip）
- 改动日志写入 `.workbuddy/memory/YYYY-MM-DD.md`

## 四、4 层测试套件职责

| 层级 | 名称 | 工具 | 工匠职责 | 守门人职责 |
|------|------|------|----------|------------|
| L1 | 单元测试 | `node --test tests/unit/` | 写单测覆盖新逻辑 | 跑 + 验证覆盖率 |
| L2 | 集成测试 | `node --test tests/integration/` | 写 API 集成测试 | 跑 + 验证 |
| L3 | E2E 演示流 | `npm run demo-test` | 写 demo-flow.test.js | **必跑**，全过才 PASS |
| L4 | 浏览器手测 | Chrome 手动 | ✗ | 跑 + 出 L4 checklist |

**硬规则**：
- L3 有 skip/fail = 守门人必须出 FAIL verdict
- "CONDITIONAL PASS" 这个词禁止使用（上次总指挥代理时埋的雷，已废除）
- 守门人 verdict 必须附完整测试输出（最后 20 行）

## 五、常见场景举例

### Q1：改价格 $4.99 → $0.99（已发生）

- 维度判定：用户感知(L2) + 业务模型(L3) = **L3 大改**
- 正确流程：总指挥 → 设计师 spec → 工匠实施 → 守门人 L1+L3 → PASS 才能 push
- 实际发生：工匠直接改 + push → **流程漏洞**
- 修复：以后 L2/L3 改动，守门人必须在 PR 里附 L1+L3 测试输出，否则不 merge

### Q2：加 Lighthouse 数据源（已发生）

- 维度判定：数据流(L3) = **L3 大改**
- 正确流程：设计师出 spec-001 → 工匠实施 → 守门人 L1+L2+L3
- 实际发生：总指挥代理了设计师 + 工匠 + 守门人 → **角色越界**
- 修复：总指挥不写代码，设计师/工匠/守门人各司其职

### Q3：修 typo "Erro" → "Error"

- 维度判定：L0
- 流程：工匠自查 → 直接 push
- 不需要守门人

## 七、违规处理

| 违规行为 | 处理 |
|---------|------|
| 工匠跳 Gate 3.5 直接 push | 守门人退回 + 重跑 L1+L3 |
| 总指挥写代码 | 守门人 FAIL + 指出角色越界 |
| 守门人"CONDITIONAL PASS" | 总指挥不认，打回重验 |
| 守门人没跑 L3 就出 PASS | 总指挥不认，罚守门人补跑 |
